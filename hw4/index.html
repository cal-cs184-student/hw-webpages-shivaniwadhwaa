Overview

In this assignment we built a real time cloth simulation from scratch. We started by constructing a mass spring system where the cloth is represented as a grid of point masses connected by three types of springs: structural, shearing, and bending. We then implemented the physics simulation using Verlet integration to update point mass positions each timestep based on forces from gravity and the spring constraints. After getting the basic simulation working we added collision detection with external objects like spheres and planes, and then tackled the harder problem of self collision using spatial hashing to efficiently detect when the cloth folds into itself. Finally we wrote a set of GLSL shader programs that run on the GPU to render the cloth with different materials and lighting models including diffuse shading, Blinn-Phong, texture mapping, bump and displacement mapping, and environment mapped reflections.

The most interesting thing we learned was how much of a difference the choice of numerical integration method makes. Verlet integration was surprisingly simple to implement but produced really stable and realistic looking results. We also found the spatial hashing for self collision really clever since the naive O(n²) approach would have been way too slow for real time use. On the shader side it was cool to see how bump and displacement mapping can produce such different results from the same texture just by changing whether you modify normals or actual vertex positions.

Part 3: Handling collisions with other objects

For sphere collisions, we first checked if the point mass was inside the sphere by seeing if its distance from the origin was less than the radius. If it was, we found the tangent point by extending the ray from the sphere's origin through the point mass out to the surface. Then we computed the correction vector from the last position to the tangent point and applied it scaled down by friction. For plane collisions, we used dot products to check which side of the plane the point mass was on currently vs the last timestep. If the signs differed, the point mass crossed the plane, so we found where it intersected and bumped it back to just above the surface using SURFACE_OFFSET, again scaling by friction.


With ks = 500, the cloth was very loose and droopy. It conformed really tightly to the sphere and the edges hung way down with a lot of folds, almost like a wet paper towel draped over a ball. With ks = 50000, the cloth was very stiff and barely conformed to the shape of the sphere at all. It kind of just sat on top with the edges sticking out more rigidly, and there were very few folds compared to the lower ks values.

Part 4: Handling self-collisions

To handle self-collisions, we implemented spatial hashing to avoid the expensive O(n²) brute force approach. We partitioned the 3D space into boxes of dimensions w*h*t and hashed each point mass's position into a unique key representing which box it belongs to. At each timestep we rebuilt this hash map so every point mass could quickly look up its neighbors. Then for each point mass we checked all candidates in the same box and if any two were closer than 2*thickness apart, we computed a correction vector to push them apart. The final correction was the average of all pairwise corrections scaled down by simulation_steps to keep things stable.


With low density like 1 g/cm², the cloth was very lightweight and fell slowly. When it hit itself it bounced around a lot and the folds were large and loose, almost like a bedsheet. With high density like 100 g/cm², the cloth was much heavier and fell faster and more aggressively. The folds were tighter and more compressed, and the cloth stacked up on itself more densely rather than spreading out.


With a low ks like 500, the cloth was very limp and folded into a lot of small tight wrinkles when it collided with itself. It almost crumpled like tissue paper. Part 5: Shaders

A shader program is a small program that runs on the GPU instead of the CPU, which makes it much faster since the GPU can run thousands of these in parallel. In our cloth simulation, we used shader programs to compute lighting and material effects in real time instead of doing expensive raytracing on the CPU. A complete shader program is made of two parts that work together: a vertex shader and a fragment shader.

The vertex shader runs first and processes each vertex of the mesh. It takes in per-vertex data like position, normal, and UV coordinates and transforms them from model space into screen space. It also passes along any data that the fragment shader will need later, like the world space position and normal, through varying variables that get interpolated across the surface of each triangle.

The fragment shader runs after rasterization and processes each pixel that ends up on screen. It takes the interpolated values that the vertex shader passed along and uses them to compute the final color of that pixel. This is where all the lighting math happens, like computing diffuse shading, specular highlights, or sampling from a texture. So the two shaders work as a pipeline where the vertex shader handles geometry and the fragment shader handles appearance, and together they produce the final rendered image.

Blinn-Phong is a lighting model that breaks down the appearance of a surface into three separate components that get added together. The ambient component is a flat constant light that gets applied everywhere on the surface equally, simulating the indirect scattered light in a scene so nothing ever appears completely black. The diffuse component depends on the angle between the surface normal and the light direction, so surfaces facing toward the light appear brighter and surfaces facing away appear darker, which gives the object its basic sense of shape and volume. The specular component simulates the shiny highlight you see on glossy surfaces, and instead of using the reflection vector directly like regular Phong shading does, Blinn-Phong uses a halfway vector between the light direction and the view direction, which is cheaper to compute and actually looks more realistic in many cases. The sharpness of the specular highlight is controlled by the exponent p, where higher values produce a tighter and more mirror-like highlight.

Bump mapping and displacement mapping both use the same height map texture to add surface detail, but they work in fundamentally different ways. Bump mapping only modifies the normal vectors used in lighting calculations, so the surface appears to have bumps and ridges when lit but the actual geometry of the mesh stays completely flat. It is essentially an illusion. Displacement mapping actually moves the vertex positions along the normal direction based on the height map, so the geometry itself changes shape. This means displacement mapping produces real silhouette changes and actual surface deformation that you can see even at the edges of the mesh.

With -o 16 -a 16, the sphere has very few vertices so displacement mapping looks blocky and jagged since there are not enough vertices to represent the fine details in the height map. Bump mapping at this resolution actually looks better in a way because the lighting trick still works regardless of mesh density. With -o 128 -a 128, displacement mapping looks dramatically better since there are enough vertices to capture the fine surface detail from the height map, producing a much more realistic and detailed result. Bump mapping looks roughly the same at both resolutions since it does not depend on vertex density at all, just the per pixel lighting calculation.
