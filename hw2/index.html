Overview

In this assignment, we built a geometric modeling engine capable of handling everything from smooth curves to complex 3D mesh manipulation. The project is divided into two main parts: generating smooth shapes using Bézier curves and surfaces, and managing detailed 3D models with the half-edge data structure. One of the most interesting things we learned is how much complexity goes into computer graphics. On paper, flipping an edge or subdividing a triangle looks simple, but in code, it becomes a high-stakes puzzle of reassigning dozens of pointers. Missing just one next or twin pointer can cause the entire model to "explode" or disappear. We also developed a much deeper appreciation for the half-edge data structure. While it is complex to set up, it enables immediate circulation around a vertex or face, which is incredibly powerful. Implementing Loop subdivision brought everything together. Seeing a sharp, blocky cube gradually morph into a smooth sphere through weighted averages and edge splits provided a clear demonstration for how modern 3D modeling software works. This project showed us that smooth surfaces are actually the result of carefully managed local geometry and consistent mathematical rules.

Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

de Casteljau's algorithm is a super clean way to find a point on a Bézier curve without having to mess with any massive, gross-looking algebraic formulas. Instead of doing heavy math, you just use linear interpolation. You take your control points, look at the lines connecting them, and pick a spot on each line based on a percentage t. Then you connect those new spots, find the same percentage t on those new lines, and just keep "peeling" it down until you're left with one single point. That last point is exactly where the curve sits.

When we actually coded this, we would've usually just set up a simple loop or a recursive function that treats the points like a shrinking list. We start with our main control points and just keep calculating the weighted average between neighbors to create a smaller set for the next round. It's way better than using the raw polynomial equations because it's super stable—it doesn't glitch out with rounding errors when the curves get complicated and it makes it really easy to just "subdivide" or split the curve if you're trying to render it or check for hits in a game.

Part 2: Bezier surfaces with separable 1D de Casteljau

So, extending de Casteljau to surfaces is actually pretty straightforward once you realize a Bézier surface is basically just a "grid of curves." Instead of a single line of control points, there's a 2D mesh. To find a point on that surface, you just need two parameters instead of one (usually called u and v) which act like your horizontal and vertical coordinates for the patch.

The way we implemented this is by breaking it down into two main steps. First, we took every row of control points in the grid and ran the standard 1D de Casteljau algorithm on them using the u parameter. This basically "squashed" each row down into a single point. Once that was done, we were left with a new, single column of intermediate points that represent a temporary Bézier curve running across the surface. Then, we just ran the algorithm one last time on that new column using the v parameter. The final point it gave us was the exact spot on the surface for those (u, v) coordinates.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

To get smooth-looking surfaces in a 3D mesh, we usually implement area-weighted vertex normals because they handle weirdly shaped triangles much better than a simple average. The basic problem is that if you have one tiny triangle and one huge triangle meeting at a corner, you want the huge one to have more "say" in which way the surface is pointing.

First, we initialized an array of zero-vectors for every vertex in the mesh. Then, we looped through every face (triangle) in the diagram. For each face, we calculated the face normal using a cross product of its edges. Instead of just adding that normal to the vertices, we multiplied it by the area of that triangle—which is easy to get since the magnitude of that same cross product is actually double the triangle's area. Once we looped through every face and accumulated those weighted vectors into the vertex "buckets," we did a final pass through the vertex array. We just normalized each vector to bring its length back to 1. This gave us a smooth gradient across the mesh that doesn't get distorted by "long, skinny" triangles, making the lighting look way more natural when it hits the image.

Part 4: Edge flip

To flip an edge between two triangles, you’re essentially re-mapping half-edges to connect the two "opposite" vertices instead of the two they currently touch. To keep the mesh from imploding, we relied on a consistency checker that runs assert() after every flip to make sure every half-edge still had a twin, and every face still formed a closed loop. When things got messy, we used visual debugging to highlight specific half-edges in the viewer, which made it way easier to spot a mismanaged pointer than staring at memory addresses.

Part 5: Edge split

For the edge split, we used a similar "collect and assign" strategy as the flip, but it’s a bit more involved since you're actually creating new mesh elements instead of just moving them. When we split an edge, we create one new vertex at the midpoint, three new half-edges, two new faces, and two new edges to subdivide the original two triangles into four. To keep the logic clean, we first grabbed all the existing elements around the two triangles and then systematically allocate the new ones, connecting the new midpoint vertex to the opposite corners. It’s basically building a small local mesh from scratch inside the hole left by the original two faces.

Part 6: Loop subdivision for mesh upsampling

Loop subdivision is essentially the "smoothing" engine of the project, and we implemented it in a four-step process to avoid messing up the geometry mid-calculation. First, we computed the new positions for all existing vertices using a weighted average of their neighbors, and then we calculated the positions for the new vertices that will eventually sit on the midpoints of every edge. We stored these as "target" positions in a temporary variable so we didn't accidentally overwrite the original data needed for other calculations. Once the math was staged, the actual mesh transformation happened. We split every edge in the original mesh—being careful to mark which edges are "new" so we didn't get stuck in an infinite loop—and then flip any new edge that connects an old vertex to a new one. Finally, we updated all vertex positions to those pre-calculated targets.
